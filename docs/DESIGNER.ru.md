# Визуальный конструктор форм – спецификация и план

Статус: волны 1-3 доставлены – этапы 0-3 (ui-схема, модель формы и операции,
панель структуры, палитра, свойства v2) плюс хуки 1-12. Бэклог хуков закрыт;
доработки по живой обратной связи – в разделе "Бэклог доработок" ниже.
Английская версия: [DESIGNER.md](DESIGNER.md). Файлы держать в синхроне.

Инструментарий получает визуальный конструктор компонентов интерфейса 1С:Элемент
(`ВидЭлемента: КомпонентИнтерфейса` – формы и свои компоненты): панель структуры,
палитру компонентов и типизированную панель свойств в VS Code, поверх уже
существующего wireframe-предпросмотра. Конструктор правит исходный `.yaml`;
главной поверхностью остаётся текстовый редактор, конструктор – контекстная
линза над ним.

## Почему такая форма

Исследование конструкторов в IDE (Flutter Property Editor, Uno Hot Design,
Android Studio Layout Editor, Delphi Object Inspector, Qt Designer, Webflow,
Figma; см. "Прототипы и источники") сходится к одному уроку: живучие и любимые
инструменты в редакторах кода – НЕ canvas со свободным позиционированием, а
триада "дерево + типизированные свойства + минимальные правки текста"; именно
canvas-дизайнеры (XAML Designer, storyboards) брошены чаще всего – медленные,
падающие, с несливаемыми диффами. Форма Элемента – буквально yaml-дерево
контейнеров, так что выигрышная форма здесь же и естественная.

## Не делаем

- Canvas со свободным позиционированием и пиксельно точный live-рендер.
  Отрисовка у платформы серверная; локальный предпросмотр остаётся честным
  wireframe: показывает структуру, а не пиксели.
- Конструктор никогда не переписывает и не переформатирует файл целиком и не
  трогает файл без явного действия пользователя (открытие формы в конструкторе
  обязано давать нулевой дифф).
- Никакой новой логики записи на стороне TypeScript. По архитектурному правилу
  репозитория модель и правки считает движок; расширение – тонкий клиент,
  который рисует деревья и применяет `WorkspaceEdit`.

## Clean-room

У платформы есть собственный веб-редактор. Этот конструктор – независимая
реализация: спецификация написана по открытой документации платформы, по
собственным извлечённым данным инструментария и по наблюдению за поведением
конструкторов в универсальных IDE. Вендорский исходный код, минифицированные
бандлы и извлечённые вендорские ассеты не читаются, не декомпилируются и не
переиспользуются. Тексты, иконки и раскладки интерфейса – свои.

## Архитектура

```
движок (Python)                         расширение VS Code (TypeScript)
---------------                         -------------------------------
ui-схема (по компонентам:               панель структуры - нативный TreeView
  свойства, типы, перечисления,         палитра          - нативный TreeView
  события, дефолты, пакеты, доки) ----> панель свойств   - WebviewView
модель формы (yaml-дерево               wireframe-предпросмотр (есть)
  со спанами узлов) + операции               |
  insert/move/remove/wrap/set   <----- операция = LSP-запрос; правки
  каждая возвращает точечные правки     применяет редактор (WorkspaceEdit)
```

- **Модель – в движке.** Новый модуль модели формы разбирает yaml компонента в
  дерево узлов со спанами исходника (включая слоты: `Команды`, `Содержимое`,
  `Подвал`, страницы, колонки) и считает операции как точечные текстовые
  правки – та же философия, что в `scaffold.py` (никакой пересериализации;
  комментарии, порядок ключей и форматирование выживают; диффы читаются как
  ручные). Поверхности – как у семейства `meta_*`: LSP-запросы для конструктора
  (только вычисление, грязные буферы через инжектированный reader; правки
  применяет редактор – сохраняется нативный undo/redo), MCP-инструменты и
  CLI-подкоманды для агентов и скриптов.
- **Нативные TreeView** для структуры и палитры: тема, клавиатура, type-ahead
  фильтр, мультивыделение и перетаскивание между двумя деревьями
  (`TreeDragAndDropController`, общий MIME) достаются бесплатно.
- **WebviewView** только там, где нужны формы ввода: панель свойств (один общий
  движок свойств для компонентов формы и объектов метаданных – заменяет обе
  текущие панели). Общий webview-хелпер (escape, nonce, CSP, инлайн JSON с
  экранированием `<`) заменяет копии в каждой панели.
- **Двусторонняя синхронизация курсора** – стержень: курсор в yaml выделяет
  узел в структуре и наполняет свойства; выбор узла ставит курсор в yaml.
  Паттерн, который у Flutter Property Editor хвалят больше всего.
- **Ограничение DnD принято:** API дерева VS Code сообщает только цель дропа,
  без позиции "между". Семантика: дроп на контейнер – вставка последним
  ребёнком; дроп на лист – вставка после него. Точный порядок – с клавиатуры
  (Alt+Вверх/Вниз и родня): по исследованию для дерева это и есть более
  точный инструмент.

## Этап 0 – фундамент (движок)

Две независимые дорожки.

**Дорожка A – ui-схема.** Расширить экстрактор stdlib, чтобы по каждому
компоненту интерфейса извлекались: пакет (`Стд::Интерфейс::*`), версия
появления и по каждому свойству – объединение типов значения (например
`Авто|Булево`, `Авто|Цвет|Url|Ссылка`), значения перечислений, признак
события (с сигнатурой обработчика), краткое описание из доков и платформенный
дефолт, где он в доках назван. Доставка – существующим версионированным
механизмом данных (данные генерируются из дистрибутива пользователя, в
репозиторий не кладутся). Новый LSP-запрос `xbsl/uiSchema` (и зеркало в MCP)
отдаёт каталог палитры и схему свойств компонента. Деградация без данных:
структура и правки работают; палитре и типизированным редакторам нужен датасет.

**Дорожка B – модель формы и операции.** Новый модуль движка: разбор yaml в
дерево узлов `{вид, имя, тип, спан, слот, дети, свойства}`; поиск узла по
смещению; операции `insert_component`, `move_node`, `remove_node`,
`wrap_node` / `unwrap_node`, `duplicate_node`, `rename_node`, `set_property` /
`reset_property` – каждая возвращает текстовые правки плюс id/спан нового
узла. Вставка учитывает правила слота (что принимает целевое свойство – один
компонент или список). Поверхности в том же изменении: LSP `xbsl/formTree`,
`xbsl/formNodeAt`, `xbsl/formEdit` и MCP/CLI (`meta_add_component`,
`meta_move_component`, ...) – агенты получают операции конструктора даром.
Полное покрытие pytest; тесты паритета поверхностей по образцу
`test_meta_surfaces.py`.

Приёмка (этап 0): гарантия round-trip (применить + откатить = байт-в-байт);
операции на корпусе реального проекта не трогают строк вне заявленных правок;
паритет трёх поверхностей.

## Этап 1 – панель структуры

Нативный TreeView "Структура", следует за активным редактором:

- Дерево = слоты и компоненты с иконками по виду, бейджи диагностик линтера
  на узлах (хук 3), type-ahead фильтр.
- Двусторонняя синхронизация выделения с yaml и панелью свойств; Ctrl+клик
  показывает yaml без переноса фокуса.
- Операции: контекстное меню + клавиатура – Alt+Вверх/Вниз, обернуть в
  Группу/Карточку (подменю типов контейнеров), убрать обёртку, дублировать,
  Del, F2-переименование, вырезать/копировать/вставить поддерево через буфер
  обмена простым yaml (работает между формами и проектами), мультивыделение
  для перемещения/удаления.
- DnD внутри дерева с отбоем недопустимых целей до дропа.
- "Сфокусироваться на поддереве" (временный корень) и фильтр "только
  именованные".

Приёмка: каждая операция – один шаг undo; синхронизация выделения до 200 мс
на форме в 1000 узлов; никаких записей в файл без явного действия.

## Этап 2 – палитра компонентов

Нативный TreeView "Палитра", вставка в текущее выделение структуры:

- Секции: Частые (счётчик использований, workspace state), Избранное, Проект
  (свои `КомпонентИнтерфейса` и вставки), дальше пакеты платформы из ui-схемы.
- Главный способ вставки – двойной клик / Enter в выделенный контейнер
  (по исследованию быстрее и точнее DnD); DnD в структуру – второй способ.
- Tooltip с описанием из доков; "Открыть документацию" ведёт в панель доков
  (хук 4).

Приёмка: вставка любого компонента каталога даёт валидный yaml, который
принимает линтер; отсутствие данных деградирует в подсказку, а не в ошибку.

## Этап 3 – панель свойств v2

Один движок свойств (WebviewView) для компонентов формы и объектов метаданных;
заменяет обе текущие панели.

- Секция "Заданные" сверху (ключи, присутствующие в yaml), ниже –
  сворачиваемые группы всех применимых свойств; поиск фильтрует по имени И по
  текущему значению.
- Индикатор задано/дефолт у строки + Reset (= удалить ключ из yaml) – самый
  переизобретаемый паттерн отрасли (жирный в WinForms, синяя точка в Hot
  Design, чипы set/default во Flutter PE, оранжевый/синий в Webflow).
- Типизированные редакторы из ui-схемы: перечисление – только dropdown;
  `Авто|Булево` – тристейт; числа; цвет (hex + образец); union-типы – парный
  редактор "Тип + Значение" (`Изображение`, `Фон`); вложенные структуры
  (`Шрифт`) – сворачиваемые подгруппы; многострочные строки; биндинги
  (`=...`) показываются как есть с выходом "Открыть в yaml" до появления
  редактора биндингов.
- Проверка значения до записи (на стороне движка), ошибка под полем.
- Выбранная строка свойства переживает переход на другой компонент того же
  типа (серийная правка – приём Delphi).

Приёмка: составные union-свойства редактируются полностью (главный пробел
текущей панели); каждая запись – минимальная текстовая правка; проверка
блокирует запись, которую линтер пометил бы ошибкой.

## Бэклог доработок (по живой обратной связи)

- **Промах активной строки yaml с выделением блока в превью.** При выборе узла в
  дереве структуры курсор yaml иногда встаёт на строку-тире (`-`) списка ПЕРЕД узлом,
  а `selectionForCursor` цепляет соседний блок (СкриншотВерх вместо Надпись =Текст) –
  похоже на off-by-one на границе элемента списка. Целить `revealOffset`/`selectionForCursor`
  в строку `Тип:` узла, а не в его тире.
- **Группировка полей по зависимостям в панели свойств.** Сейчас есть только индикатор
  слота (полоса + бейдж); полноценная группировка зависимых полей не сделана.

Параллельные дорожки; каждый этап выходит обычным минорным релизом движка и
расширения вместе.

- Волна 1 (параллельно): дорожка A (ui-схема) + дорожка B (модель формы) –

- Волна 2 (параллельно, после A/B): структура; палитра; свойства v2.
  Хуки 3 и 4 входят в эти этапы.
- Волна 3 (параллельно): хук 1 (события) + хук 2 (панель "Данные") – две
  самые сильные RAD-привычки.
- Волна 4+: остальные хуки по мере надобности.

## Риски

- **Нет датасета** (публичная установка без сгенерированных данных): палитра
  и типизированные редакторы деградируют; структура и правки обязаны работать.
- **DnD-API дерева** не выражает позицию "между" – компенсируется
  клавиатурным порядком; не воевать с API через webview-дерево.
- **Цена состояния webview**: панель свойств живёт на `getState`/`setState`,
  без `retainContextWhenHidden`.
- **Большие формы**: построение дерева и поиск узла – инкрементально
  (переиспользовать разбор открытого буфера LSP, дебаунс синхронизации).
- **Эпоха двух панелей**: пока этап 3 не заменил панель свойств метаданных,
  они сосуществуют; старую панель не развивать.
