Справочник собран из самого инструмента – это то же, что показывает `xbsl --help`, только на одной странице и целиком.

Без команды `xbsl` проверяет указанные пути: это режим по умолчанию, и его опции перечислены в первом блоке. Остальные команды адресуют другие части инструментария.

Язык вывода переключается флагом `--lang`, переменной `XBSL_LANG` или берётся из локали системы.

## Без команды: проверка исходников

Линтер исходников 1С:Элемент (пары .yaml/.xbsl).

```bash
usage: xbsl [пути] [опции]        (без команды: проверка исходников)
       xbsl <команда> [опции]
```

**Аргументы**

| Параметр | Описание |
|---|---|
| `paths` | файлы или каталоги для проверки |

**Параметры**

| Параметр | Описание |
|---|---|
| `-h, --help` | показать эту справку и выйти |
| `--select ID/ГРУППА/ТИР` | проверять только эти правила (через запятую или повтором флага: id, группа – часть id до '/' (напр. style) – или буква тира A/B/C/D) |
| `--ignore ID/ГРУППА/ТИР` | исключить эти правила (через запятую или повтором флага: id, группа или буква тира) |
| `--enable ID/ГРУППА/ТИР` | добавить выключенные по умолчанию правила ПОВЕРХ стандартного набора (`--select` набор заменяет); формы значений те же |
| `--baseline ФАЙЛ` | гасить находки, замороженные в файле базлайна (создаётся `--write-baseline`); новые находки выводятся как обычно |
| `--write-baseline ФАЙЛ` | вместо отчёта записать все текущие находки в файл базлайна (заморозить долг; пути в файле – относительно его каталога) |
| `--fix` | исправить механические находки на месте (хвостовые пробелы, типографские символы, переводы строк) и вывести оставшиеся; правит только однозначно |
| `--jobs N` | процессов для файловых правил: 0 – авто (включается на больших прогонах), 1 – последовательно, N – явное число воркеров |
| `--list-rules` | вывести список правил и выйти |
| `--where` | показать корень данных Элемента (путь, источник, версии) и выйти |
| `--element-version ВЕРСИЯ` | версия данных Элемента (по умолчанию – последняя из бандла) |
| `--data-dir КАТАЛОГ` | корень данных Элемента (каталог с index.json); также env XBSL_DATA_DIR |
| `--lang {ru,en}` | язык вывода линтера (по умолчанию: env XBSL_LANG / локаль системы / ru) |
| `--format {text,json,codeclimate}` | формат вывода: text (по умолчанию), json (машиночитаемый: diagnostics + summary) или codeclimate (отчёт GitLab Code Quality – виджет в merge request) |
| `--stdin` | проверить один буфер из stdin (для интеграции с редактором); вид файла и путь в позициях задаёт `--filename` |
| `--index` | вместо проверки вывести JSON-индекс проекта (объекты, методы, компоненты форм) для навигации в редакторе; путь – корень проекта |
| `--filename ИМЯ` | имя проверяемого буфера при `--stdin` (напр. Форма.xbsl); расширение задаёт вид файла |
| `--version` | показать версию и выйти |

**Команды**

| Команда | Описание |
|---|---|
| `lint <пути>` | проверить исходники – то же, что без команды |
| `lsp` | сервер LSP для редактора |
| `mcp` | сервер MCP для агента |
| `web` | веб-панель |
| `templates` | шаблоны кода: list, export, import, save |
| `self-update` | обновить xbsl распаковкой колеса с PyPI |

Опции команды: xbsl &lt;команда&gt; `--help`. Опции выше относятся к режиму проверки.

## `xbsl lsp`

LSP-сервер xbsl (stdio)

```bash
usage: xbsl-lsp [-h] [--project-root PROJECT_ROOT] [--select SELECT] [--ignore IGNORE]
                [--enable ENABLE] [--baseline BASELINE] [--templates TEMPLATES]
                [--data-dir DATA_DIR] [--lang {ru,en}]
```

**Параметры**

| Параметр | Описание |
|---|---|
| `-h, --help` | показать эту справку и выйти |
| `--project-root PROJECT_ROOT` | корень исходников (абсолютный или относительно папки воркспейса) |
| `--select SELECT` | только эти правила (через запятую) |
| `--ignore IGNORE` | исключить правила (через запятую) |
| `--enable ENABLE` | включить правила поверх набора по умолчанию |
| `--baseline BASELINE` | файл базлайна (абсолютный или относительно папки воркспейса) – исключённые находки гасятся; отсутствующий файл не ошибка, он появится с первым исключением |
| `--templates TEMPLATES` | файл шаблонов кода (абсолютный или относительно папки воркспейса) – дополняет встроенный набор, одноимённые шаблоны замещает |
| `--data-dir DATA_DIR` | корень данных Элемента (папка с index.json) |
| `--lang {ru,en}` | язык текста замечаний |

MCP-сервер xbsl (stdio): линт, документация Элемента и скаффолдинг метаданных как инструменты агента.

```bash
usage: xbsl-mcp [-h]
```

**Параметры**

| Параметр | Описание |
|---|---|
| `-h, --help` | показать эту справку и выйти |

## Скаффолдинг метаданных

Команды создают и правят исходники: объекты, поля, маршруты, методы, формы, подсистемы. Каждая печатает результат в JSON и проверяет линтером то, что записала; `--dry-run` считает изменения, не трогая файлы.

### `xbsl new-project`

```bash
usage: xbsl new-project [-h] [--representation REPRESENTATION] [--version VERSION]
                        [--compatibility COMPATIBILITY] [--subsystem SUBSYSTEM] [--library]
                        [--dry-run]
                        root vendor name
```

**Аргументы**

| Параметр | Описание |
|---|---|
| `root` | каталог, в котором появится пара поставщик/имя (обычно .) |
| `vendor` | поставщик – первая часть пространства имён проекта |
| `name` | имя проекта; так же называется его каталог |

**Параметры**

| Параметр | Описание |
|---|---|
| `-h, --help` | показать эту справку и выйти |
| `--representation REPRESENTATION` | представление проекта в интерфейсе (по умолчанию – имя) |
| `--version VERSION` | версия проекта, три числа (по умолчанию 1.0.0) |
| `--compatibility COMPATIBILITY` | версия платформы, с которой совместим проект (по умолчанию 9.0) |
| `--subsystem SUBSYSTEM` | имя первой подсистемы (по умолчанию Основное) |
| `--library` | создать библиотеку (ВидПроекта: Библиотека), а не приложение |
| `--dry-run` | показать изменения (с текстами файлов), ничего не записывая |

### `xbsl new-object`

```bash
usage: xbsl new-object [-h] [--scope SCOPE] [--environment ENVIRONMENT] [--access ACCESS]

                       directory kind name
```

**Аргументы**

| Параметр | Описание |
|---|---|
| `directory` | каталог подсистемы, в котором создать объект |
| `kind` | вид объекта на языке проекта: Справочник, Документ, ВиртуальнаяТаблица, ...; неизвестный вид перечислит доступные |
| `name` | имя объекта |

**Параметры**

| Параметр | Описание |
|---|---|
| `-h, --help` | показать эту справку и выйти |
| `--scope SCOPE` | область видимости; по умолчанию платформенная ВПодсистеме |
| `--environment ENVIRONMENT` | окружение – для ОбщийМодуль и Структура |
| `--access ACCESS` | способ доступа: у HttpСервис пишется в Разрешения.Вызов, у объектов данных – в Разрешения.ПоУмолчанию (отдельные права ставит set-access) |
| `--routes ROUTES` | маршруты HttpСервис: "GET /, POST /, GET /{id}" |
| `--report REPORT` | описание отчёта (JSON: source, rows, columns, measures) |
| `--dry-run` | показать изменения (с текстами файлов), ничего не записывая |

### `xbsl form-edit`

```bash
usage: xbsl form-edit [-h] [--parent PARENT] [--slot SLOT] [--type TYPE] [--name NAME]
                      [--node NODE] [--nodes ID[,ID...]] [--new-parent NEW_PARENT]
                      [--container CONTAINER] [--new-name NEW_NAME] [--before BEFORE]
                      [--after AFTER] [--key KEY] [--value VALUE] [--value-yaml VALUE_YAML]
                      [--fragment FRAGMENT] [--fragment-file ФАЙЛ] [--new-type NEW_TYPE]
                      [--dry-run]
                      yaml_path операция
```

**Аргументы**

| Параметр | Описание |
|---|---|
| `yaml_path` | yaml формы |
| `операция` | insert, insert-fragment, move, move-nodes, remove, remove-nodes, wrap, unwrap, duplicate, rename, set-property, reset-property, property-add, property-retype, property-remove, property-rename |

**Параметры**

| Параметр | Описание |
|---|---|
| `-h, --help` | показать эту справку и выйти |
| `--parent PARENT` | id узла-контейнера (insert/insert-fragment) |
| `--slot SLOT` | слот детей: Содержимое, Страницы, Колонки, ... (insert/move) |
| `--type TYPE` | Тип нового компонента (insert) или свойства (property-add) |
| `--name NAME` | Имя нового компонента (insert), обёртки (wrap) или свойства секции Свойства (property-*) |
| `--node NODE` | id узла операции (move/remove/wrap/unwrap/duplicate/rename/set-property/reset-property) |
| `--nodes ID[,ID...]` | id узлов пачковой операции (move-nodes/remove-nodes): через запятую или повтором флага; порядок не важен |
| `--new-parent NEW_PARENT` | id нового контейнера (move/move-nodes) |
| `--container CONTAINER` | Тип контейнера-обёртки (wrap) |
| `--new-name NEW_NAME` | новое Имя узла (rename) или свойства (property-rename); для rename без флага Имя удаляется |
| `--before BEFORE` | id соседа: вставить/переместить ПЕРЕД ним |
| `--after AFTER` | id соседа: вставить/переместить ПОСЛЕ него |
| `--key KEY` | имя свойства узла (set-property/reset-property) |
| `--value VALUE` | скалярное значение или биндинг (set-property) |
| `--value-yaml VALUE_YAML` | составное значение готовым yaml-фрагментом (set-property) |
| `--fragment FRAGMENT` | yaml-блок компонента или нескольких – список "-" или блоки подряд (insert-fragment) |
| `--fragment-file ФАЙЛ` | файл с yaml-блоком компонента (insert-fragment, вместо `--fragment`) |
| `--new-type NEW_TYPE` | новый Тип свойства (property-retype) |
| `--dry-run` | показать изменения (с текстами файлов), ничего не записывая |

### `xbsl form-handlers`

```bash
usage: xbsl form-handlers [-h] [--node NODE] [--key KEY] [--method METHOD] [--signature SIGNATURE]
                          [--dry-run]
                          yaml_path
```

**Аргументы**

| Параметр | Описание |
|---|---|
| `yaml_path` | yaml формы |

**Параметры**

| Параметр | Описание |
|---|---|
| `-h, --help` | показать эту справку и выйти |
| `--node NODE` | id узла (создание обработчика; без `--node`/`--key` – список методов модуля) |
| `--key KEY` | ключ события узла: ПриНажатии, ПослеСоздания, ... |
| `--method METHOD` | имя метода-обработчика (по умолчанию &lt;Имя узла&gt;&lt;Ключ&gt;; существующий метод – только привязка в yaml) |
| `--signature SIGNATURE` | сигнатура события из ui-схемы, напр. "(Кнопка, СобытиеПриНажатии)-&gt;ничто" (без флага ищется в локальных данных) |
| `--dry-run` | показать изменения (с текстами файлов), ничего не записывая |
