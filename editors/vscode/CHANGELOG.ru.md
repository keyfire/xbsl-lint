# История изменений

[English](https://github.com/keyfire/xbsl-lint/blob/main/editors/vscode/CHANGELOG.md) · **Русский**

## 0.15.2

- Открытие документа по правилу или по ссылке теперь прокручивает к нужному разделу (якорю), а не к
  началу страницы: у `naming/module-suffix` – к разделу общих правил наименования, у `project/version`
  – к разделу "Версия" и т.д. Заголовки разделов в документации сохраняют якоря.

## 0.15.1

- Ссылка на документ стандарта из диагностики теперь открывает раздел **внутри VS Code** (панель
  "Документация" и позиционирование дерева), а не внешний сайт.

## 0.15.0

- Ссылка на документ стандарта прямо из диагностики. У правил, реализующих платформенные стандарты
  (`naming/*`, `project/*`, `query/in-subquery-composite`), значок правила в "Проблемах" стал
  кликабельной ссылкой на страницу стандарта, а лампочка на находке предлагает **XBSL: документация
  по правилу** – открывает документ в панели "Документация" и позиционирует на нём дерево
  "Содержание".

## 0.14.1

- README дополнен разделом про панель документации (в 0.14.0 он был только в списке изменений):
  описаны дерево "Содержание", поиск, страница с картинками и первоисточником, документация по
  символу правым кликом и команды `xbsl.docs.*`.

## 0.14.0

- Новый вид **Документация** на панели 1С:Элемент: дерево "Содержание" справки Элемента, поиск по
  документации и показ страницы (тип, его методы, свойства, параметры) с картинками и ссылкой на
  первоисточник. Правый клик на переменной или типе в `.xbsl` – "XBSL: документация по символу" –
  открывает нужную страницу. Данные даёт LSP-сервер линтера (нужен `xbsllint` >= 0.12.0 с базой
  документации, собранной `extract_docs.py`); в обычном режиме (CLI) панель сообщает, что
  документация доступна в LSP-режиме.

## 0.13.0

- Новая группа правил **project** в настройках: свойства проекта по стандарту "Заполнение свойств
  проекта" – `Поставщик` и `Имя` как идентификаторы с прописной буквы, заполненные `Представление`
  и `ПредставлениеПоставщика`, версия тремя числами `A.B.C`.
- Новое правило в группе **query** (нужен `xbsllint` >= 0.11.0): `В` с подзапросом по полю
  составного типа (`Строка|Число`) – по стандарту платформы такое условие пишется через
  `СУЩЕСТВУЕТ`, на большинстве СУБД `В` с подзапросом реализовано неэффективно.
- Новая группа правил **naming** в настройках (нужен `xbsllint` >= 0.11.0): имена элементов проекта
  по стандарту платформы "Имена элементов проекта" – число по виду элемента (справочники во
  множественном числе, перечисления в единственном), буква `ё` и подчёркивания, аббревиатуры одним
  словом, вид элемента внутри имени, постфикс окружения у общего модуля, незаполненное
  Представление. Все двенадцать правил – предупреждения; группу целиком можно приглушить или
  выключить выпадающим списком, как остальные.

## 0.12.0

- Новый контейнер **1С:Элемент** на панели действий (своя иконка): элементы проекта сгруппированы
  по виду – справочники, общие модули, HTTP-сервисы и т.д., у каждой группы своя иконка (codicon).
  Форма объекта / форма списка вложена под свой объект-владелец; формы без владельца – в отдельный
  раздел **Общие формы**. Пара yaml + xbsl объекта – одна строка: контекстное меню открывает описание
  (yaml), модуль (xbsl), модуль объекта или предпросмотр формы.
- Корень дерева – **проект**; его контекстное меню открывает модуль приложения. Объекты раскрываются
  в поддеревья – **Реквизиты / Измерения / Ресурсы / Табличные части / Формы**; новое поле можно
  добавить в Реквизиты / Измерения / Ресурсы (действие **+** спрашивает имя и вставляет минимальную
  заготовку со свежим Ид, затем показывает её).
- Поведение клика: общий модуль открывает свой xbsl, форма – предпросмотр, поле – переход в yaml.
- Ещё поддеревья: **Значения** перечисления (+ добавить), **Параметры** работы клиента (+ добавить),
  **Шаблоны URL** HTTP-сервиса с их методами (только просмотр). У корня проекта серым – Поставщик\Имя.
- Дерево показывает создаваемые классы объектов, даже когда их в проекте ещё нет (справочник,
  документ, перечисление, регистр сведений/накопления, общий модуль, HTTP-сервис, параметры работы
  клиента). В корне категории – по-видовое действие **Добавить &lt;класс&gt;** (Добавить справочник,
  Добавить документ ...): спрашивает имя и подсистему (папку), пишет минимальный валидный yaml (свежий
  Ид; парный xbsl у модульных видов) и открывает его. Созданный объект не деплоится, пока вы его не
  доделаете, – невалидный всплывёт только при деплое.
- **Подсистемы**: ветка **Подсистемы** под проектом (открыть подсистему или **Добавить подсистему** –
  папку с файлом подсистемы). У корня проекта – **Отбор по подсистеме** (мультивыбор) и **Снять
  отбор**; активный отбор виден серым рядом с проектом.
- Ещё создаваемые классы: **структура, клиентское событие, фрагмент командного интерфейса** и
  самостоятельная **общая форма** (в разделе Общие формы) тоже получили своё действие добавления.
- Редактируемая **панель свойств** справа: клик по объекту или полю открывает её (модули и формы –
  через контекстный пункт **Свойства**). Скалярные свойства правятся на месте (undo работает); Ид и
  вид элемента – только для чтения; коллекции остаются в дереве.
- Поддерево **Поля** с добавлением у структур; **Добавить форму объекта** справочнику/документу
  (создаёт форму и регистрирует её в объекте, если её ещё нет); **Удалить объект** (с подтверждением;
  удаляет файлы объекта, ссылки не обновляются).
- **Табличные части** справочника/документа теперь тоже группа с добавлением – **Добавить табличную
  часть** создаёт секцию со стартовым реквизитом; у самой табличной части есть **Добавить реквизит
  табличной части**.
- **Статус git** показан на строках дерева (как в Проводнике): объекты, формы, подсистемы и проект
  несут SCM-декорацию файла (цвет и бейдж), сохраняя иконку вида.
- Основная кнопка в предпросмотре формы теперь нативного жёлтого цвета платформы (`#fd0`, тёмный
  текст) вместо синего.
- **Тип** реквизита / измерения / ресурса / поля правится комбобоксом в панели свойств: примитивы,
  ссылки `<Объект>.Ссылка?` и перечисления проекта предлагаются подсказками, а ввести можно и любой
  другой тип.
- Строкоспецифичные свойства (`Многострочная`) в панели свойств показываются только для типа `Строка`
  и убираются при смене типа на другой.
- Группа **Стандартные реквизиты** у справочников/документов показывает предопределённые реквизиты
  (Наименование/Код, Номер/Дата), даже когда их нет в yaml; правка свойства в панели материализует
  запись в `Реквизиты` (без Ид, как у стандартного реквизита).
- В **статус-баре** – версия/время сборки расширения, версия xbsllint и режим дополнения (обычный
  CLI-индекс / LSP): удобно понять, какая сборка реально запущена.
- В блоке `Запрос{...}` дополнение после `<Таблица>.` предлагает **поля таблицы** (стандартные поля,
  реквизиты, табличные части), а не члены объекта – и в обычном режиме, и в LSP-сервере (в индекс
  `xbsllint` добавлены реквизиты объекта).
- **LSP-режим включён по умолчанию** (`xbsl.lsp.enabled`): он даёт hover и дополнение по типам. Если
  линтер стоит без extra `[lsp]`, расширение молча продолжает в прежнем CLI-режиме – окно с ошибкой
  больше не всплывает, а статус-бар показывает режим, который работает на самом деле.
- **Дополнение по типам** (LSP-режим): к таблице запроса можно обращаться по алиасу (`ИЗ Акция КАК А`
  → `А.` даёт поля Акции); переменная цикла по результату запроса (`для С из Результат` → `С.`) даёт
  колонки выборки; переменная известного типа (`пер Список = новый Массив<Строка>()` → `Список.`) и
  типы/глобали stdlib (`КонтекстДоступа.`) дают свои члены. Разбор идёт по токенам, поэтому ключевые
  слова понимаются в обеих формах (`пер`/`var`, `новый`/`new`). Свойства и методы показаны раздельно:
  у метода свой значок, вставляется он со скобками. Имя в области видимости сильнее одноимённого типа
  (объявлен `Список` – речь о его типе, а не о компоненте `Список`). Нужен `xbsllint` >= 0.10.0.
- Метки дерева метаданных (категории и поддеревья) теперь следуют языку интерфейса – английскому или
  русскому – как и остальное расширение.
- Клик по объекту, полю, модулю или форме в дереве открывает исходник слева (описание yaml, а у
  код-видов – `.xbsl`) и панель свойств / предпросмотр формы справа, переиспользуя уже открытые колонки
  и панели вместо новых; панель свойств при перекликивании выходит на передний план в своей колонке.
- Дерево синхронизировано с редактором: активный объект / модуль / форма выделяется в дереве (пока
  оно на виду), а только что добавленный объект, поле, подсистема или форма – показывается и
  выделяется сразу после создания.
- Дерево можно группировать **по классам объектов** (по умолчанию) или **по подсистемам** (объекты
  под своими папками-подсистемами, подсистемы вложены по папкам) – переключатель в заголовке дерева;
  выбор запоминается.

## 0.10.1

  каркас формы 1С:Элемента в webview – вложенные группы, надписи, поля с подписями и биндингами,

## 0.9.0

- Новая секция настроек **Группы правил**: выпадающий список на каждый тип находок (code, yaml, style,
  typography, whitespace, encoding, structure, form) – оставить собственные уровни правил, показывать
  всю группу одним уровнем или выключить её (правила тогда не запускаются, а не просто скрыты). Больше
  не нужно вручную вписывать идентификаторы в `xbsl.rules` для частых случаев; `xbsl.rules` остаётся

- Лампочка "Настроить правило..." получила ярлык "Настроить группы правил..." в новую секцию.

## 0.8.0

- Новая команда *XBSL: деплой на стенд (elemctl)* (`xbsl.deploy`), она же кнопка-облачко в заголовке
  редактора у `.xbsl`: запускает `elemctl deploy` – сборка из исходников, загрузка, применение,
  перезапуск и проверка фактического применения – терминальной задачей, после диалога подтверждения с

  заданный `xbsl.projectRoot` передаётся как `--project-dir`. Предлагает установить elemctl, если его нет.
- Английский README теперь показывает английские заголовки команд (двуязычие с 0.6.1).

## 0.4.0

- Quick Fix для механических находок: лампочка на исправимой диагностике (хвостовые пробелы,
  типографские символы – длинное тире → среднее, `…` → `...`, кудрявые кавычки) применяет ровно ту
  правку, которую сообщил линтер. Нужен линтер, отдающий правки в JSON (`xbsllint` ≥ 0.7.1).
- Source-действие *исправить все* (`source.fixAll.xbsl`) чинит все исправимые находки файла разом;
  подключите его в `editor.codeActionsOnSave` для исправления по сохранению. Правки применяются только
  к тому тексту, на котором были вычислены (снимок с версией), так что устаревшая правка не смещается.

## 0.2.0

- Диагностика по воркспейсу: сохранение любого `.xbsl`/`.yaml` запускает полный прогон линтера по
  папке воркспейса (с задержкой, по одному за раз, устаревшие прогоны отменяются), принося проектные
  правила (`code/unknown-type`, `yaml/unknown-type`, ...) в редактор. Результат прогона замещает
  диагностику каждого файла; быстрый `--stdin`-линт владеет только редактируемым буфером.
- Новые настройки: `xbsl.workspaceLint` (включено по умолчанию) и `xbsl.workspaceLintTimeout`
  (60000 мс; по истечении прогон останавливается и пишется в панель вывода XBSL).
- Команда *XBSL: проверить весь проект* переиспользует ту же механику и хранилище результатов.
